# Инженерные правила

Правила для кода и процесса изменений в монорепозитории Электротехника.

## Код и структура

- **Небольшие файлы**: предпочитать несколько сфокусированных модулей одному «божественному» файлу.
- **Маленькие компоненты**: одна ответственность; составные экраны собирать из частей, а не раздувать один компонент.
- **Ясные границы**: код публичного приложения не должен импортировать внутренние модули supplier и наоборот; общее — через `packages/*`.
- **Без скрытой магии**: избегать чрезмерной рефлексии, глобальных синглтонов и неочевидных сайд-эффектов при импорте.

## Абстракции

- Не вводить **лишние слои** (лишние интерфейсы, абстрактные фабрики) без трёх конкретных потребителей или явного требования.
- Не создавать **преждевременные** общие пакеты «на все случаи жизни».

## Предметная область

- **Не моделировать домен** (сущности БД, DTO, бизнес-типы) до появления явного скоупа в задаче.
- Не добавлять схемы таблиц и миграции «на будущее».

## Качество и инструменты

- TypeScript в **strict**-режиме там, где это задано конфигами пакетов.
- Перед коммитом полезно прогонять из корня: `pnpm lint`, `pnpm typecheck`, `pnpm build` (по ситуации).
- **ESLint** и **Prettier** — обязательная база; локальные отключения прав — с комментарием и причиной.

## Документация

- Значимые архитектурные изменения — **ADR** и правка [architecture.md](architecture.md) / [README.md](../README.md) при необходимости.
- Если меняется способ работы агентов — обновить [AGENTS.md](../AGENTS.md) и при необходимости [llm-playbook.md](llm-playbook.md).

## Импорты

- Внутренние пакеты: `@electrotech/<name>`.
- В приложениях Next — алиас `@/*` для локального кода приложения, не для подмены общих пакетов.
