# Архитектура монорепозитория

## Структура

```
apps/
  web/        — публичный Next.js (App Router)
  supplier/   — Next.js для поставщиков/админки (App Router)
  api/        — NestJS, единая точка HTTP API
packages/
  ui/              — общий минимальный UI
  utils/           — общие утилиты без домена
  config-eslint/   — общий ESLint (Next / type-checked TS)
  config-typescript/
  config-tailwind/
docs/
  adr/        — архитектурные решения
```

## Зачем несколько приложений

- **Изоляция зон продукта**: публичная часть и внутренняя/поставщическая часть развиваются и деплоятся независимо.
- **Снижение риска утечек**: отдельный кодовый базис для supplier упрощает контроль доступа и конфигурации на этапе эксплуатации.
- **Масштабирование команд**: разные области можно вести параллельно при сохранении общих пакетов.

## Стратегия общих пакетов

- **Конфиг-пакеты** (`config-*`) — единые правила TypeScript, ESLint и базовый пресет Tailwind.
- **UI** — только переиспользуемые примитивы; страницы и фичи остаются в приложениях.
- **utils** — узкие функции без бизнес-семантики; рост пакета должен быть осознанным.

Приложения подключают внутренние пакеты через `workspace:*` и при необходимости `transpilePackages` в Next.js.

## Docker и локальная разработка

- **docker compose** поднимает PostgreSQL, Redis и три сервиса приложений в dev-режиме.
- Контекст сборки Docker — **корень монорепозитория**, чтобы установка зависимостей через pnpm была корректной.
- Для работы без Docker можно запускать `pnpm dev`; базы данных тогда поднимаются отдельно или через частичный compose.

## Как безопасно добавлять модули

1. **В приложении** — новые маршруты и фичи в `apps/<app>/src` в рамках границ приложения.
2. **В API** — новые Nest-модули под `apps/api/src`, без смешивания с UI.
3. **В общий пакет** — только при реальном повторном использовании; избегать «библиотеки на вырост».
4. Обновить **документацию** и при значимых решениях — **ADR**.

## Публичный API каталога (сервис)

Приложение `apps/api` подключается к PostgreSQL (`DATABASE_URL`). Модули каталога:

- `GET /products`, `GET /products/:slug` — список и карточка товара (минимальные поля для витрины).
- `GET /categories` — плоский список категорий для фильтров в `apps/web`.
- `GET /suppliers`, `GET /suppliers/:slug` — список и карточка поставщика.

При пустой таблице товаров выполняется однократное заполнение демо-данными (сид).
