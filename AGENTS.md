# AGENTS.md — правила работы в репозитории Электротехника

Этот файл предназначен для человеческих разработчиков и для **LLM-агентов** (включая Cursor). Соблюдение правил снижает архитектурный дрейф и ускоряет итерации.

## Назначение репозитория

Монорепозиторий **electrotech** — основа для продукта «Электротехника». Сейчас в нём только каркас: Next.js-приложения, NestJS API с health-эндпоинтами, общие конфиг-пакеты и минимальный UI. **Не подменяйте** этот каркас вымышленной предметной моделью без явной задачи.

## Зафиксированные архитектурные решения

- **Монорепозиторий** на pnpm workspaces + Turborepo.
- **Три приложения** с чёткими границами:
  - `apps/web` — публичный лендинг и публичный сервис (когда появятся).
  - `apps/supplier` — зона поставщика и администрирования.
  - `apps/api` — единая HTTP-оболочка для бэкенда.
- **Общий UI** только в `packages/ui`; приложения не копируют дизайн-систему целиком в свои деревья без причины.
- **Инфраструктура для локальной разработки** — Docker Compose (PostgreSQL, Redis, сервисы приложений).

Менять эти решения только через осознанное обсуждение и новую запись в `docs/adr/`.

## Границы приложений

| Приложение | Разрешено                                              | Запрещено без отдельной задачи                    |
| ---------- | ------------------------------------------------------ | ------------------------------------------------- |
| `web`      | Публичные маршруты, маркетинг, публичный UX            | Логика кабинета поставщика, внутренние админ-флоу |
| `supplier` | Кабинет поставщика, внутренние инструменты             | Публичный лендинг как основной дом                |
| `api`      | HTTP API, интеграции, фоновые задания (когда появятся) | Встраивание UI или статики Next                   |

## Границы пакетов

- `packages/config-*` — только конфигурация инструментов; не добавлять туда runtime-код продукта.
- `packages/utils` — мелкие перекрёстные помощники **без** доменных типов и без «фреймворка внутри фреймворка».
- `packages/ui` — презентационные примитивы; не тащить сюда запросы к API и глобальное состояние приложения.

## Перед написанием кода

1. Прочитать этот файл и [docs/engineering-rules.md](docs/engineering-rules.md).
2. При затрагивании архитектуры — [docs/architecture.md](docs/architecture.md) и релевантные ADR в [docs/adr/](docs/adr/).
3. При продуктовом контексте — [docs/product-context.md](docs/product-context.md) (высокий уровень, без выдуманных сущностей).

## Figma MCP

- **Figma — источник правды по UI**, когда для экрана есть макет.
- Перед реализацией экрана: просмотреть нужный фрейм/компонент через Figma MCP, зафиксировать отступы, типографику, состояния.
- **Не выдумывать** крупные визуальные отклонения; при неясности — задокументировать допущение в PR/комментарии или в задаче, а не «додумывать молча».
- Соблюдать токены и сетку из макета; общие примитивы выносить в `packages/ui` по мере появления реальных повторов.

Подробнее: [docs/figma-workflow.md](docs/figma-workflow.md).

## Против архитектурного дрейфа

- Не добавлять технологии, не согласованные с корневым стеком (см. README), без ADR и обновления документации.
- Не создавать параллельные «вторые» папки с тем же смыслом (`lib2`, `components-v2`) без явной миграции.
- Не подключать ORM, очереди, поиск, провайдеры auth **до** явной задачи и дизайна в ADR при необходимости.

## Спекулятивные абстракции

- **Запрещено** вводить слои «на будущее»: лишние репозитории, фабрики, event bus, generic CRUD без конкретного use case.
- **Разрешено** простое дублирование до появления третьего повтора или явного требования.

## Доменная сложность

- Не вводить сущности, DTO, схемы БД и типы предметной области, пока в задаче это **явно** не запрошено.
- Health/readiness и переменные окружения — не повод для полноценной модели каталога или пользователей.

## Изменение структуры репозитория

- Новое приложение — только с обновлением README, `docs/architecture.md`, новым ADR при существенном сдвиге.
- Новый общий пакет — описать в `docs/architecture.md` и в README (кратко).
- Скрипты в корне должны оставаться согласованными с реальными `package.json` воркспейсов.

## Итог для агента

Делайте минимальный дифф, сохраняйте стиль существующего кода, читайте документацию до рефакторинга и не расширяйте предметную область без формулировки в задаче.
